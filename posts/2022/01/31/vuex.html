Vuex

# 概念

Vuex，它是一个程序里面的状态管理模式，它是集中式存储所有组件状态的小仓库.更新状态的唯一方法是提交mutation

# 解决了什么问题

  • 多个组件依赖同一状态时，对于多层嵌套的组件的传参会比较繁琐
  • 不同组件的行为需要变更同一状态

# 注册store

import Vue from 'vue';
import Vuex from 'vuex';

Vue.use(Vuex);

const store = new Vuex.Store({
    state: {
        // 定义一个name，以供全局使用
        name: '张三',
        // 定义一个number，以供全局使用
        number: 0,
        // 定义一个list，以供全局使用
        list: [
            { id: 1, name: '111' },
            { id: 2, name: '222' },
            { id: 3, name: '333' },
        ]
    },
});

export default store;

# state

const Counter = {
  template: `<div>{{ count }}</div>`,
  computed: {
    count () {
      return this.$store.state.count
    }
  }
}

每当this.$store.state.count变化时，都会重新计算，并且触发更新相关联的DOM

# getters

const store = new Vuex.Store({
  state: {
    todos: [
      { id: 1, text: '...', done: true },
      { id: 2, text: '...', done: false }
    ]
  },
  getters: {
    doneTodos: state => {
      return state.todos.filter(todo => todo.done)
    }
  }
})

//组件里使用
computed: {
  doneTodosCount () {
    return this.$store.getters.doneTodosCount
  }
}

getter是有缓存机制的，类似于computed

# muations

改变Vuex的state不能直接修改，而要通过muations或actions进行修改

muations进行同步操作的修改，不能在里面进行异步操作

const store = new Vuex.Store({
  state: {
    count: 1
  },
  mutations: {
    increment (state, n) { // n为参数，可设置，可不设置，此参数也称为“载荷”
      // 变更状态
      state.count++
    }
  }
})
//使用
// 使用
this.$store.commit('increment', 10)

# action

action中提交的是mutions，而不是直接改变状态

action可以包含任意异步操作

const store = new Vuex.Store({
  state: {
    count: 0
  },
  mutations: {
    increment (state) {
      state.count++
    }
  },
  actions: {
      // Action 函数接受一个与 store 实例具有相同方法和属性的 context 对象
     incrementAsync (context , n) { // 可传“载荷” n
       setTimeout(() => {
         context.commit('increment') 
       }, 1000)
      }
  }
})

//使用
// 执行
// 以载荷形式分发
store.dispatch('incrementAsync', {
  amount: 10
})

// 以对象形式分发
store.dispatch({
  type: 'incrementAsync',
  amount: 10
})

# Module

由于使用单一状态树，应用的所有状态会集中到一个比较大的对象，当应用变得庞大时，store对象就可能变得臃肿。

所以Vuex允许我们将store分割成模块（module）。每个模块拥有自己的state``mutation``action``getter。

const moduleA = {
  state: () => ({ ... }),
  mutations: { ... },
  actions: { ... },
  getters: { ... }
}

const moduleB = {
  state: () => ({ ... }),
  mutations: { ... },
  actions: { ... }
}

const store = new Vuex.Store({
  modules: {
    a: moduleA,
    b: moduleB
  }
})

store.state.a // -> moduleA 的状态
store.state.b // -> moduleB 的状态

对于模块内部的mutation和getter，接收的第一个参数是模块的局部状态对象。

const moduleA = {
  state: () => ({
    count: 0
  }),
  mutations: {
    increment (state) {
      // 这里的 `state` 对象是模块的局部状态
      state.count++
    }
  },

  getters: {
    doubleCount (state) {
      // 这里的 `state` 对象是模块的局部状态
      return state.count * 2
    }
  }
}

同样，对于模块内部的action，局部状态通过context.state暴露，根节点状态是context.rootState,

模块内部的getter也是

const moduleA = {
  // ...
  actions: {
    incrementIfOddOnRootSum ({ state, commit, rootState }) {
      if ((state.count + rootState.count) % 2 === 1) {
        commit('increment')
      }
    }
  }
}

# 命名空间

默认情况下，模块内部的action、mutation、getter是注册在全局命名空间，如果多个模块中的action、mutation的命名是一样的，那么提交mutation、action时，将会触发所有模块中命名相同的mutation和action

这样有太多的耦合，可以通过添加namespaced：ture，使其成为命名空间模块

export default{
    namespaced: true,
    state,
    getters,
    mutations,
    actions
}

在之后组件的使用中，携带上命名空间即可，action这些也是这样做

themeData() {
      return this.$store.getters['themeModule/themeData']
    },
    proName() {
      return this.$store.getters['productModule/proName']
  	},

如果想提交全局的mutation或action，携带上一个参数root:true即可

this.$store.dispatch('actionA', null, { root: true })
this.$store.commit('mutationA', null, { root: true })

const store = new Vuex.Store({
    strict:true,
})

# 实现

# 为什么每个组件都能访问到store

这个问题和每个组件都能访问到router和route一样。

这是因为在进行插件注册(**Vue,use()**)的时候，调用了Vuex中的install方法，在install中，进行了全局的混入（mixin），在每个组件实例化之前，会先判断组件是否为根组件，如果是根组件，就把options中的store，赋值给实例的$store；如果不是根组件，则让组件往上寻找其父组件的$store。

最后使得每个组件都拥有了$store

# install方法

// vuex.js
let Vue;

// install方法设置，是因为Vue.use(xxx)会执行xxx的install方法
const install = (v) => { // 参数v负责接收vue实例
    Vue = v;
    // 全局混入
    Vue.mixin({
        beforeCreate() {
            if (this.$options && this.$options.store) {
                // 根页面，直接将身上的store赋值给自己的$store，
                这也解释了为什么使用vuex要先把store放到入口文件main.js里的根Vue实例里
                this.$store = this.$options.store;
            } else {
                // 除了根页面以外，将上级的$store赋值给自己的$store
                this.$store = this.$parent && this.$parent.$store;
            }
        },
    })
}

# 为什么state/getter具有响应式

  • 在Store构造函数中，会把getters的方法包装一层，收集到computed中。这也是为什么getters拥有缓存的原因，它利用了computed的缓存原理

  • 在每次getters被触发的时候，会直接返回store._vm中的属性（这个store._vm是一个Vue的实例，具体看下面）

  store.getters = {}
  //_wrappedGetters会收集所有getters方法
  const wrappedGetters = store._wrappedGetters
  const computed = {}
  Object.keys(wrappedGetters).forEach(key => {
    const fn = wrappedGetters[key]
    // use computed to leverage its lazy-caching mechanism
    computed[key] = () => fn(store)
    Object.defineProperty(store.getters, key, {
      get: () => store._vm[key]
    })
  })
  • 将state和getters的值分别传入new Vue中初始化成data和computed，使其具有响应式的特性。这里的computed就是上面的getters
 store._vm = new Vue({
   data: { state },
   computed
 })
