# nextTick

# 概念

在下次DOM更新循环结束之后执行的延迟回调，在修改数据之后立即使用该方法，获取更新后的DOM

可以用一个简单的例子理解：

<template>
  <div class="hello">
    <p ref="p_ref">
      当前姓名:<b>{{ name }}</b>
    </p>
    <button @click="changeName">改变名称</button>
  </div>
</template>

<script>
export default {
  name: "Demo",
  data() {
    return {
      name: "张三",
    };
  },
  methods: {
    changeName() {
      this.name = "李四";
      console.log(this.$refs["p_ref"].innerHTML);
    },
  },
};
</script>

改变了name的值之后立即打印dom节点的内容

当前姓名:<b>张三</b>

这里并没有拿到最新的数据，用nextTick改写

<template>
  <div class="hello">
    <p ref="p_ref">
      当前姓名:<b>{{ name }}</b>
    </p>
    <button @click="changeName">改变名称</button>
  </div>
</template>

<script>
export default {
  name: "Demo",
  data() {
    return {
      name: "张三",
    };
  },
  methods: {
    changeName() {
      this.name = "李四";
      this.$nextTick(() => {
        console.log(this.$refs["p_ref"].innerHTML);
      });
    },
  },
};
</script>

可以打印出最新的name值——李四

这就是nextTick的作用：帮助我们在改变组件中属性之后，立刻拿到渲染以后的dom节点对象

# 原理

在Vue中，当页面数据改变的时候并不会立刻去进行UI的渲染，而是会把这些UI渲染的任务放到一个异步队列中。

  • 异步：只要侦听到数据变化，vue即开启一个队列并缓存当前eventloop内所有的数据变更
  • 批量：每一次eventloop内将所有的watch都存入queue内，且一个watch只会被推入queue一次并去重，这样可以避免不必要的浏览器重绘

通俗一点讲，在执行this.name="李四"的时候，就会触发Watch的更新，watch会把自己放入到一个队列中。

用队列并去重的原因是比如多个数据变更就更新视图多次的话，性能会比较差。

# 触发变更与微任务队列

微任务是在一次循环中一直取一直取，直到清空microtask队列，而宏任务则是一次循环取一次。因为当触发数据的setter时，vue就会在microtask中推出一个回调，等宏任务执行完毕之后再去执行这些回调，这些回调的作用就是更新DOM视图

mounted() {
    console.log(this.$el.clientHeight) // 0
    this.name = "沐华"
    console.log(this.$el.clientHeight) // 0
    this.$nextTick(() => {
      console.log(this.$el.clientHeight) // 18
});

也就是执行this.name="沐华"时，任务队列可以理解成[flushSchedulerQueue]

然后下一行执行打印，由于会更新视图的flushSchedulerQueue在队列中还没执行，所以无法拿到更新后的视图。

接着执行nextTick，又会添加一个异步任务，这个时候队列就变成了[flushSchedulerQueue,fn]

等同步任务执行完毕之后，接着按照顺序执行任务队列里面的任务，第一个任务执行完毕之后就会更新视图，后面自然能拿到更新后的视图

# 源码解析

# 触发变更

当我们触发某个数据的set后，他的set函数会通知闭包中的Dep，Dep则会调用它管理的所有Watch对象，触发Wacth对象的update。

update代码

/*调度者接口，当依赖发生改变的时候进行回调 */
  update () {
    ...
      /*异步推送到观察者队列中，下一个tick时调用。*/
      queueWatcher(this)
  }

当数据发生变化的时候，会调用queueWatcher(this)

queueWatcher

 /*将一个观察者对象push进观察者队列，在队列中已经存在相同的id则该观察者对象将被跳过，除非它是在队列被刷新时推送*/
export function queueWatcher (watcher: Watcher) {
  /*获取watcher的id*/
  const id = watcher.id
  /*检验id是否存在，已经存在则直接跳过，不存在则标记哈希表has，用于下次检验*/
  if (has[id] == null) {
    has[id] = true
    if (!flushing) {
      queue.push(watcher)
    } else {
      // if already flushing, splice the watcher based on its id
      // if already past its id, it will be run next immediately.
      let i = queue.length - 1
      while (i >= 0 && queue[i].id > watcher.id) {
        i--
      }
      queue.splice(Math.max(i, index) + 1, 0, watcher)
    }
    // queue the flush
    if (!waiting) {
      waiting = true
      // 因为每次派发更新都会引起渲染，所以把所有 watcher 都放到 nextTick 里调用
      nextTick(flushSchedulerQueue)
    }
  }
}

这里参数 flushSchedulerQueue 方法就会被放入事件循环，主线程任务的行完后就会执行这个函数，对 watcher 队列排序、遍历、执行 watcher 对应的 run 方法，然后 render，更新视图

# 环境判断

因为宏任务耗费时间是大于微任务的，所以优先使用微任务，判断顺序如下

  • Promise

  • MutationObserver

  • setImmediate

  • setTimeout

export let isUsingMicroTask = false // 是否启用微任务开关
const callbacks = [] // 回调队列
let pending = false // 异步控制开关，标记是否正在执行回调函数

// 该方法负责执行队列中的全部回调
function flushCallbacks () {
  // 重置异步开关
  pending = false
  // 防止nextTick里有nextTick出现的问题
  // 所以执行之前先备份并清空回调队列
  const copies = callbacks.slice(0)
  callbacks.length = 0
  // 执行任务队列
  for (let i = 0; i < copies.length; i++) {
    copies[i]()
  }
}


let timerFunc // 用来保存调用异步任务方法
// 判断当前环境是否支持原生 Promise
if (typeof Promise !== 'undefined' && isNative(Promise)) {
  // 保存一个异步任务
  const p = Promise.resolve()
  timerFunc = () => {
    // 执行回调函数
    p.then(flushCallbacks)
    // ios 中可能会出现一个回调被推入微任务队列，但是队列没有刷新的情况
    // 所以用一个空的计时器来强制刷新任务队列
    if (isIOS) setTimeout(noop)
  }
  isUsingMicroTask = true
} else if (!isIE && typeof MutationObserver !== 'undefined' && (
  isNative(MutationObserver) ||
  MutationObserver.toString() === '[object MutationObserverConstructor]'
)) {
  // 不支持 Promise 的话，在支持MutationObserver的非 IE 环境下
  // 如 PhantomJS, iOS7, Android 4.4
  let counter = 1
  const observer = new MutationObserver(flushCallbacks)
  const textNode = document.createTextNode(String(counter))
  observer.observe(textNode, {
    characterData: true
  })
  timerFunc = () => {
    counter = (counter + 1) % 2
    textNode.data = String(counter)
  }
  isUsingMicroTask = true
} else if (typeof setImmediate !== 'undefined' && isNative(setImmediate)) {
  // 使用setImmediate，虽然也是宏任务，但是比setTimeout更好
  timerFunc = () => {
    setImmediate(flushCallbacks)
  }
} else {
  // 以上都不支持的情况下，使用 setTimeout
  timerFunc = () => {
    setTimeout(flushCallbacks, 0)
  }
}

环境判断结束之后就会返回一个延迟回调函数timerFunc

然后进入核心的nextTick

# nextTick()

主要逻辑如下：

  • 把传入的回到函数放进回调队列callbacks
  • 执行保存的异步任务timerFunc，就会遍历执行对应的微任务
export function nextTick (cb?: Function, ctx?: Object) {
  let _resolve
  // 把回调函数放入回调队列
  callbacks.push(() => {
    if (cb) {
      try {
        cb.call(ctx)
      } catch (e) {
        handleError(e, ctx, 'nextTick')
      }
    } else if (_resolve) {
      _resolve(ctx)
    }
  })
  if (!pending) {
    // 如果异步开关是开的，就关上，表示正在执行回调函数，然后执行回调函数
    pending = true
    //这里就会执行我们的视图更新和获取新dom了
    timerFunc()
  }
  // 如果没有提供回调，并且支持 Promise，就返回一个 Promise
  if (!cb && typeof Promise !== 'undefined') {
    return new Promise(resolve => {
      _resolve = resolve
    })
  }
}

# 总结

Vue 在更新 DOM 时是异步执行的。只要侦听到数据变化，Vue 将开启一个队列，并缓冲在同一事件循环中发生的所有数据变更。如果同一个 watcher 被多次触发，只会被推入到队列中一次。这种在缓冲时去除重复数据对于避免不必要的计算和 DOM 操作是非常重要的。然后，在下一个的事件循环“tick”中，Vue 刷新队列并执行实际 (已去重的) 工作。Vue 在内部对异步队列尝试使用原生的 Promise.then、MutationObserver 和 setImmediate，如果执行环境不支持，则会采用 setTimeout(fn, 0) 代替。
